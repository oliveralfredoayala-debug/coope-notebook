# Flujo de agentes · Trabajos en equipo

## Secuencia base
1. Team Work Orchestrator
2. Collaborative Pedagogy Architect
3. Case, Content & Evidence Auditor
4. Group Dynamics & Facilitation Agent
5. Cooperative Law Core Authority, si aplica
6. Interaction & Experience Builder
7. Accessibility, Mobile & Inclusive Participation Reviewer
8. Functional, Export & Integration QA

## Retornos
- Si una interacción limita pensamiento propio, vuelve a Pedagogía.
- Si una etapa no genera conversación real, vuelve a Dinámica grupal.
- Si el caso no sostiene la consigna, vuelve a Contenido.
- Si hay competencia dudosa, vuelve a Legal.
- Si una corrección accesible altera la actividad, vuelve a Experiencia y Pedagogía.
- Si falla guardado, exportación o navegación, vuelve a Experiencia.
- Si cambia el nombre final, vuelve a Integración/QA.

## Regla
No avanzar con hallazgos bloqueantes. Los desacuerdos entre agentes se conservan en el reporte y los resuelve el Orquestador con prioridad: exactitud y seguridad → accesibilidad → pedagogía → funcionamiento → presentación.

## Gate jurídico reforzado

- R2 y R3 requieren revisión ADVANCED.
- R3 no pasa a Experiencia sin aprobación expresa del Cooperative Law Core Authority.
- Toda corrección implementada vuelve al agente jurídico para comprobar que el HTML no alteró el sentido.
- Un hallazgo R4 detiene publicación y exportación.


## Gate de sala en vivo y Sandbox

Después de implementación y antes de QA final:

1. `Live Room Sandbox & Configuration Inspector` normaliza la configuración.
2. Verifica evocación desde `lnl`, estrategia `invoke` y ausencia de duplicación.
3. Si hay bloques legales, exige sistema `kb-purple` y aprobación de `Cooperative Law Core Authority`.
4. QA confirma que el panel muestra la configuración efectiva y no expone respuestas ni secretos.
